## 0. Ключевые решения

### Архитектура
- **API сервер:** VPS на Beget (Python FastAPI + SQLite/PostgreSQL). Полный контроль, бот и API на одном сервере
- **База данных:** Supabase (PostgreSQL) — хранение данных, бесплатно до 500 МБ
- **Фронтенд:** GitHub Pages (статика: HTML + CSS + JS)
- **Связь:** Фронтенд → API на VPS → Supabase
- **Бот:** На том же VPS, работает 24/7

### Оффлайн-режим
- Приложение работает без интернета через localStorage
- При появлении интернета — автоматическая синхронизация с сервером
- Конфликты: серверная версия приоритетнее (если данные изменились на двух устройствах)

### Оплата
- **Старт:** ЮКасса (нужен ИП или самозанятость)
- **Позже:** Telegram Stars, Робокасса
- Архитектура оплаты — универсальная, легко добавить новый провайдер

### Тарифы (могут измениться)
| Тариф | Цена | Для кого | Ключевые функции |
|-------|------|----------|-----------------|
| **Бесплатный (MVP)** | 0 ₽ | Обычные люди | 2 кошелька, фиксированные категории, 10 операций/день, базовая аналитика |
| **Pro** | 149 ₽/мес | Фрилансеры, мастера | До 10 кошельков, свои категории, безлимит операций, экспорт Excel, таблица расходов |
| **Макс** | 500 ₽/мес | Бизнес, ИП | Всё из Pro + несколько направлений бизнеса, все новые функции |

**Важно:** Тарифная модель — не финальная. Структура базы данных и API должны позволять легко менять тарифы, добавлять новые, менять лимиты без переписывания кода. Тарифы хранятся в отдельной таблице, не захардкожены.

## 1. Схема работы (как всё связано)

```
Пользователь (Telegram)
    ↓
Telegram Mini App (GitHub Pages)
    ↓ (запросы с JWT-токеном)
API сервер (VPS Beget, FastAPI)
    ↓ (SQL-запросы)
Supabase (PostgreSQL)

Telegram Bot (VPS Beget, aiogram)
    ↓ (уведомления, отчёты)
Пользователь (Telegram чат)
```

**Почему VPS как прокси, а не напрямую к Supabase:**
1. Проверка Telegram initData (HMAC) — нельзя делать на клиенте (нужен BOT_TOKEN)
2. Проверка лимитов и тарифов — нельзя доверять клиенту
3. Бот работает на этом же VPS — одна точка
4. Оплата через ЮКассу — вебхуки приходят на VPS
5. Генерация Excel — тяжёлая операция, делается на сервере

## 3. API-эндпоинты (FastAPI на VPS)

**Базовый URL:** `https://api.mycash.app/v1/` (или IP VPS на старте)
**Авторизация:** JWT-токен в заголовке `Authorization: Bearer <token>`
**Версионирование:** `/v1/` — чтобы при изменениях старые клиенты не ломались

### 3.1 Авторизация

| Метод | URL | Описание |
|-------|-----|----------|
| POST | `/v1/auth/telegram` | Авторизация через Telegram initData |

**Подробная логика:**
1. Фронтенд отправляет `{ init_data: tg.initData }`
2. Сервер парсит initData, извлекает user_id, hash
3. Сервер вычисляет HMAC-SHA256 с BOT_TOKEN, сравнивает с hash
4. Если подпись невалидна → 401 Unauthorized
5. Ищет пользователя по telegram_id в базе
6. Если нет — создаёт: users + 2 кошелька + 17 демо-операций
7. Генерирует JWT (payload: user_id, telegram_id, tariff, exp: 24h)
8. Возвращает: `{ token, user: { id, first_name, tariff, tariff_until } }`

**Безопасность:** BOT_TOKEN хранится ТОЛЬКО на сервере. Никогда не передаётся клиенту.

---

### 3.2 Операции

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `/v1/operations?period=month&year=2026&month=4` | Список за период |
| POST | `/v1/operations` | Создать (с проверкой лимита) |
| PUT | `/v1/operations/{id}` | Редактировать |
| DELETE | `/v1/operations/{id}` | Удалить |

**POST — проверка лимита:**
```python
# Подсчёт операций за сегодня (надёжнее чем счётчик)
today_count = SELECT COUNT(*) FROM operations 
    WHERE user_id = :uid AND created_at::date = CURRENT_DATE AND is_demo = false

if today_count >= tariff_limits.max_daily_operations and user.tariff == 'free':
    return 429, "Лимит 10 операций в день. Обновите тариф."
```

**POST — тело запроса:**
```json
{
    "type": "expense",
    "amount": 1500,
    "category": "Продукты",
    "wallet_id": "uuid",
    "comment": "Пятёрочка",
    "date": "2026-04-09T10:00:00Z"
}
```

**GET — ответ:**
```json
{
    "operations": [...],
    "summary": {
        "income": 96200,
        "expense": 63249,
        "balance": 32951
    }
}
```

## 5. Оффлайн-режим (подробно)

### Как работает:
1. При запуске: `api.js` пытается авторизоваться на сервере
2. **Если есть интернет:** загружает данные с сервера, сохраняет копию в localStorage
3. **Если нет интернета:** берёт данные из localStorage (последняя копия)
4. Все операции (создание, удаление, редактирование) сохраняются в localStorage
5. Помечаются как `synced: false`
6. При появлении интернета — POST `/v1/user/sync` отправляет все несинхронизированные

### Конфликты:
- Если операция изменена и на клиенте, и на сервере — **серверная версия побеждает**
- Удалённые на сервере операции — удаляются и из localStorage
- После синхронизации — полная перезагрузка данных с сервера

### Индикация:
- В шапке: иконка "облако" (синхронизировано) или "облако перечёркнутое" (оффлайн)
- При оффлайн-сохранении: "Сохранено локально. Синхронизируется при подключении"

## 6. Что меняется во фронтенде

### 6.1 Новый файл `api.js`
- Авторизация (POST `/v1/auth/telegram`)
- Все CRUD-операции
- Синхронизация оффлайн-данных
- Обработка ошибок (429 лимит, 401 не авторизован, 500 сервер)

### 6.2 Изменения в `app.js`
| Функция | Было (localStorage) | Станет (API + оффлайн) |
|---------|---------------------|----------------------|
| init() | Storage.load() | API.auth() → API.getOperations() → fallback localStorage |
| saveOperation() | operations.unshift() | API.createOperation() → fallback localStorage + sync queue |
| deleteOperation() | operations.filter() | API.deleteOperation() → fallback localStorage |
| updateBalance() | Подсчёт в JS | API.getWallets() → cached |
| renderDashboard() | Подсчёт в JS | API.getDashboard() → cached |

### 6.3 Кэширование
- Данные кэшируются в localStorage на 5 минут
- При переключении вкладок — берутся из кэша если свежие
- При добавлении/удалении операции — кэш сбрасывается

### 6.4 Индикатор загрузки
- Спиннер при первой загрузке
- Skeleton-loading при переключении периодов
- "Нет соединения" с кнопкой "Повторить"

## 8. Миграция (localStorage → Supabase)

1. При первом входе после обновления — проверяем localStorage
2. Если есть данные и на сервере пусто → POST `/v1/user/migrate`
3. Сервер создаёт все операции в базе
4. Очищаем localStorage (кроме UI-настроек)
5. Уведомление: "Данные перенесены на сервер 🎉"

## 9. Мониторинг и логирование

- **Логи API:** Все запросы логируются (endpoint, user_id, время, статус)

- **Бэкапы:** Ежедневный дамп базы (pg_dump) на VPS → копия на внешнее хранилище

## 11. Админ-панель (через Telegram-бота)

Админ-функции доступны только для Telegram ID владельца (Виктор). Проверка: `if telegram_id != ADMIN_ID: return`.

**ADMIN_ID** хранится в `.env`.

### Команды админа

| Команда | Что делает |
|---------|-----------|
| `/admin` | Главное меню админки (inline-кнопки) |
| `/stats` | Общая статистика: пользователей, операций, платежей за сегодня/неделю/месяц |
| `/users` | Последние 10 зарегистрированных пользователей |
| `/user {telegram_id}` | Инфо о пользователе: тариф, операций, кошельков, дата регистрации |
| `/set_tariff {telegram_id} {tariff}` | Вручную выдать тариф (free/pro/max) |
| `/payments` | Последние 10 платежей (статус, сумма, метод) |
| `/payments {telegram_id}` | Платежи конкретного пользователя |
| `/broadcast {текст}` | Отправить сообщение всем пользователям |
| `/limits` | Показать текущие лимиты тарифов |
| `/set_limit {tariff} {поле} {значение}` | Изменить лимит (например: max_daily_operations) |
| `/block {telegram_id}` | Заблокировать пользователя |
| `/unblock {telegram_id}` | Разблокировать |

### Пример `/stats`:
```
📊 Статистика MyCash

Пользователей: 147
  Free: 130 | Pro: 15 | Макс: 2

Сегодня:
  Новых: 3
  Операций: 89
  Платежей: 1 (149 ₽)

Неделя:
  Новых: 18
  Операций: 412
  Платежей: 4 (746 ₽)
```

### Пример `/user 123456789`:
```
👤 Пользователь #123456789

Имя: Виктор
Тариф: Pro (до 15.05.2026)
Зарегистрирован: 09.04.2026
Кошельков: 3
Операций: 47
Последняя активность: сегодня
```

### Безопасность:
- Команды работают ТОЛЬКО для ADMIN_ID
- Для обычных пользователей — бот отвечает как обычно (/start, /help)
- ADMIN_ID хранится в .env, не в коде

### Добавление в таблицу users:
Поле `is_blocked` (boolean, default false) — заблокированный пользователь не может использовать API.

### Реализация бота-админки: Этап 3 (вместе с ботом)

## 11.2 Веб-админка (admin.mycash.app)

Полноценная панель управления в браузере. Доступ по паролю или через Telegram OAuth.

### Страницы:

| Страница | Что показывает |
|----------|---------------|
| **Дашборд** | Графики: пользователи/день, операции/день, доход/день. Ключевые метрики |
| **Пользователи** | Таблица: ID, имя, тариф, операций, дата регистрации, статус. Поиск, фильтры |
| **Пользователь** | Детально: все данные, кошельки, последние операции, платежи. Кнопки: сменить тариф, заблокировать |
| **Платежи** | Список всех платежей: дата, пользователь, сумма, статус, метод |
| **Тарифы** | Редактирование лимитов, цен, включение/выключение функций |
| **Рассылка** | Форма отправки сообщения всем/части пользователей |
| **Логи** | Последние ошибки API, подозрительная активность |

### Технология:
- Простой HTML + JS (как основное приложение) или FastAPI + Jinja2 шаблоны
- Защита: логин/пароль или Telegram OAuth
- Хостинг: на том же VPS

### Реализация: Этап 5 (после запуска основных функций)

**Приоритет:** Сначала бот-админка (быстро, Этап 3), потом веб-админка (красиво, Этап 5).
